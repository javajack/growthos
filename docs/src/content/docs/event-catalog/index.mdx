## Purpose

This catalog provides **ready-to-use event definitions** for the most common business verticals. Each domain page includes event names, properties, lifecycle stage mapping, and volume classification.

Use this as your primary reference when instrumenting your product with GrowthOS. Instead of inventing event names from scratch, find your domain below, copy the schemas that apply, and customize the properties to match your data model.

<Aside type="tip">
Every event in this catalog follows the canonical structure defined in the [Event Schema](/growthos/api/events/) page. The catalog builds on that foundation with domain-specific naming and property recommendations.
</Aside>

## How to Use This Catalog

<Steps>
1. **Find your domain** (or the closest match) in the directory below.

2. **Start with Cross-Domain Universal events** â€” these are required for every GrowthOS installation regardless of vertical.

3. **Add domain-specific events** from your vertical's page. Each page organizes events by growth loop stage (Acquire, Activate, Engage, Monetise, Advocate).

4. **Customize properties** â€” the schemas here are a starting template. Add properties that matter to your business and remove ones that do not apply.

5. **Follow naming conventions** from the [Event Schema](/growthos/api/events/) page to keep your taxonomy clean and automations reliable.
</Steps>

## Naming Conventions

A quick recap of the rules that apply to every event in this catalog.

<CardGrid>
  <Card title="snake_case everywhere" icon="seti:json">
    All event names and property keys use `snake_case`. No camelCase, no PascalCase, no kebab-case.
  </Card>
  <Card title="object.action pattern" icon="list-format">
    Domain events follow `object.action` naming â€” e.g., `order.completed`, `subscription.upgraded`, `feature.activated`.
  </Card>
  <Card title="$ prefix = system only" icon="setting">
    Events starting with `$` (e.g., `$page_view`, `$session_start`) are auto-tracked by the SDK. Never use the `$` prefix for custom events.
  </Card>
  <Card title="Custom events = no prefix" icon="pencil">
    Developer-defined events are free-form within the naming rules. No `$` prefix, no reserved words.
  </Card>
</CardGrid>

## Volume Classification

Each event in the catalog is tagged with a volume badge indicating expected frequency and recommended processing strategy.

| Badge | Meaning | Processing Guidance |
|---|---|---|
| <Badge text="High" variant="caution" /> | Auto-tracked or very frequent user actions | Use batching and sampling; avoid storing raw payloads long-term |
| <Badge text="Medium" variant="note" /> | Regular user actions | Standard processing pipeline; index key properties |
| <Badge text="Low" variant="success" /> | Rare events (conversions, lifecycle changes) | Fully processed; store complete payloads; trigger automations |
| <Badge text="Low (admin)" variant="default" /> | Operational and internal events | Admin dashboards only; not exposed to end-user analytics |

<Aside type="note">
Volume classifications are guidelines based on typical SaaS and e-commerce traffic patterns. Your actual volumes will depend on product usage. Adjust batching and retention policies accordingly.
</Aside>
