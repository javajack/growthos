These events are **required for every GrowthOS installation** regardless of your business domain. They cover the core user lifecycle that every product shares: authentication, billing, support, notifications, consent, and growth signals.

<Aside type="tip">
Events marked with <Badge text="High" variant="caution" /> are auto-tracked by the GrowthOS SDK where possible. You only need to instrument them manually if you are using the server-side API directly.
</Aside>

## Authentication and Identity

11 events covering signup, login, security, and account management.

| Event | Key Properties | Volume | Description |
|---|---|---|---|
| `user.signed_up` | `method`, `source`, `referral_code`, `plan` | <Badge text="Low" variant="success" /> | User creates a new account |
| `user.signed_in` | `method` (password / sso / magic_link / oauth), `provider` | <Badge text="High" variant="caution" /> | User authenticates successfully |
| `user.signed_out` | `session_duration` | <Badge text="High" variant="caution" /> | User ends their session |
| `user.password_reset_requested` | `method` | <Badge text="Low" variant="success" /> | User requests a password reset link |
| `user.password_reset_completed` | `method` | <Badge text="Low" variant="success" /> | Password reset flow completed |
| `user.email_verified` | `verification_method` | <Badge text="Low" variant="success" /> | User confirms their email address |
| `user.mfa_enabled` | `mfa_method` (totp / sms / webauthn) | <Badge text="Low" variant="success" /> | Multi-factor authentication turned on |
| `user.mfa_disabled` | `mfa_method` | <Badge text="Low" variant="success" /> | Multi-factor authentication turned off |
| `user.profile_updated` | `fields_changed` (array) | <Badge text="Medium" variant="note" /> | User modifies profile fields |
| `user.deleted` | `reason`, `feedback` | <Badge text="Low" variant="success" /> | User account permanently deleted |
| `user.impersonated` | `admin_id`, `target_user_id` | <Badge text="Low (admin)" variant="default" /> | Admin impersonates a user for debugging |

</Aside>

## Lifecycle Signals

6 events that power growth loop analytics, churn prediction, and reactivation workflows.

| Event | Key Properties | Volume | Description |
|---|---|---|---|
| `lifecycle.first_value_moment` | `action`, `time_to_value_seconds`, `session_number` | <Badge text="Low" variant="success" /> | User reaches their first "aha" moment |
| `lifecycle.activation_milestone` | `milestone_name`, `milestone_index`, `days_since_signup` | <Badge text="Low" variant="success" /> | User completes a defined activation milestone |
| `lifecycle.expansion_signal` | `signal_type`, `score` | <Badge text="Medium" variant="note" /> | Behavior suggests readiness for upsell |
| `lifecycle.churn_risk_detected` | `risk_score`, `signals` (array), `days_inactive`, `usage_trend` | <Badge text="Low (admin)" variant="default" /> | System flags a user as churn risk |
| `lifecycle.reactivation_detected` | `days_since_last_active`, `trigger` | <Badge text="Low" variant="success" /> | Previously inactive user returns |
| `lifecycle.advocacy_triggered` | `trigger` (nps_promoter / review_posted / referral_sent) | <Badge text="Low" variant="success" /> | User exhibits advocacy behavior |

<Aside type="tip">
Lifecycle signals are the foundation for the [Churn Prediction](/growthos/phase-4/churn-prediction/) and [Contact Scoring](/growthos/phase-2/contact-scoring/) modules. Instrument these early, even if you are not using those modules yet.
</Aside>

## Implementation Example

A typical onboarding flow tracking signup, identity resolution, and the first value moment.

```javascript
import GrowthOS from '@growthos/js';

const gos = GrowthOS.init('YOUR_WRITE_KEY');

// 1. Track signup
gos.track('user.signed_up', {
  method: 'email',
  source: 'landing_page',
  referral_code: 'REF_abc123',
  plan: 'free'
});

// 2. Identify the user (merges anonymous_id with user_id)
gos.identify('usr_42', {
  email: 'jamie@example.com',
  name: 'Jamie Chen',
  created_at: '2025-06-15T09:00:00Z',
  plan: 'free'
});

// 3. Track activation milestone
gos.track('lifecycle.activation_milestone', {
  milestone_name: 'created_first_project',
  milestone_index: 1,
  days_since_signup: 0
});

// 4. Track first value moment
gos.track('lifecycle.first_value_moment', {
  action: 'generated_first_report',
  time_to_value_seconds: 340,
  session_number: 1
});

// 5. Track subscription creation
gos.track('subscription.created', {
  plan_name: 'pro',
  billing_interval: 'monthly',
  mrr_cents: 4900,
  trial: true
});
```

<Aside type="caution">
Do not put PII (email, phone, full name) in event `properties`. Store PII as traits via `identify()` instead. See the [Event Schema best practices](/growthos/api/events/) for details.
</Aside>
